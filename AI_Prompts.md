# AI Prompts 关键交互记录

说明：这份文档是根据项目开发过程整理的关键交互摘要，用来说明我在做这个 AI 编程项目时，怎样借助 AI 拆需求、写代码、改设计和解决问题。内容保留了当时的提问意图、AI 给出的方向，以及最后在项目中的落地结果。

## 项目背景

我想做一个本地可运行的 LLM 标注工具。主要目标是把 Excel 数据导入系统，配置 Prompt、知识库、错题集和标注方案，然后批量调用模型完成标注，并在工作台里看到 TP/FP、准确率、F1 等指标。

项目最后形成了三步式工作流：

- 开始使用工具
- 数据集与方案管理
- 标注工作台

技术栈选择：

- 后端：FastAPI
- 数据库：SQLite
- 前端：HTML / CSS / JavaScript
- 表格：Tabulator
- 实时更新：SSE
- 用户扩展：`user_hooks/`

## 交互 1：确定项目整体方向

我的提问：

```text
我想做一个本地运行的 LLM 标注工具，主要用于导入 Excel 数据，然后配置 Prompt 调模型批量标注。帮我拆一下功能模块和技术方案，我是新手，尽量用简单稳定的方案。
```

AI 帮助：

AI 建议先做一个本地工作台，把系统拆成“场景管理、数据集导入、Prompt 管理、知识库、错题集、标注方案、标注工作台”几个模块。技术方案建议使用 FastAPI + SQLite + 静态前端，减少部署复杂度。

落地结果：

项目采用了本地启动方式，入口是 `python3 run.py`，后端挂载前端静态页面，数据库使用 SQLite。整体功能写进了 `README.md` 和 `prd-business.md`。

## 交互 2：设计三步式页面流程

我的提问：

```text
这个工具页面不要太复杂，我希望用户一步一步用。帮我设计一个简单页面流程，先介绍工具，再管理数据和方案，最后进入标注工作台。
```

AI 帮助：

AI 建议做成三步式单页应用，顶部固定显示当前步骤：开始使用工具、数据集与方案管理、标注工作台。这样比赛演示时路线清楚，用户也容易理解。

落地结果：

前端实现了三步导航，页面逻辑拆在 `frontend/pages/start.js`、`frontend/pages/manage.js`、`frontend/pages/workbench.js`。

## 交互 3：设计数据结构

我的提问：

```text
Excel 的列可能很多，每个业务场景的数据字段也不一样。SQLite 里怎么存比较简单？我不想一开始就做复杂的动态建表。
```

AI 帮助：

AI 建议每个场景建立一张独立数据表，用 `raw_data` 保存整行 JSON。数据集表里保存列结构 `column_schema`，前端根据列结构动态展示。

落地结果：

后端支持场景、数据集和场景数据表。导入 Excel 后，数据进入 SQLite，前端只请求当前页数据，降低大表加载压力。

## 交互 4：实现 Excel 导入

我的提问：

```text
帮我写一个 Excel 导入接口，上传文件后读取表头和每一行，保存到 SQLite。后面前端要能分页看数据。
```

AI 帮助：

AI 给了 FastAPI 上传接口、Excel 解析、字段 schema 保存和数据行入库的实现思路，并提醒导入后前端只展示摘要和分页数据。

落地结果：

项目实现了数据集导入能力，README 中也记录了“Excel 数据导入后写入 SQLite，前端只请求当前页数据”的性能策略。

## 交互 5：Prompt 占位符规则

我的提问：

```text
Prompt 里面要引用当前行的数据，比如工单名称、API 内容、知识库。怎么设计占位符比较稳？Prompt 里还会写 JSON 示例，怕和模板语法冲突。
```

AI 帮助：

AI 建议使用 `[[row.列名]]`、`[[knowledge]]`、`[[error_sets]]` 这种占位符，避免和 JSON 里的 `{}` 冲突。渲染时只解析 `[[...]]`。

落地结果：

项目采用了 `[[...]]` 规则。文档中写了示例：

```text
工单名称：[[row.工单名称]]
API 内容：[[row.API Part 1]]
知识库：[[knowledge]]
错题集：[[error_sets]]
```

## 交互 6：自定义 Prompt 初始化

我的提问：

```text
有些方案可能会选多个 Prompt，比如质检员、复核员。不同 Prompt 可能要组成不同角色。这个怎么做成可扩展？
```

AI 帮助：

AI 建议把 Prompt 初始化独立出来，自动替换适合简单场景，自定义初始化适合多 Prompt、多角色和公司内部 messages 格式。

落地结果：

项目在 `user_hooks/prompt_init_methods.py` 中预留了自定义 Prompt 初始化方法，可以返回 `{角色名: Prompt对象}`。

## 交互 7：设计用户扩展入口

我的提问：

```text
这个项目以后要接公司内部模型，我不想每次都改后端主代码。能不能留一个简单入口，让业务开发只改一个地方？
```

AI 帮助：

AI 建议把业务扩展放到 `user_hooks/` 里，后端只调用统一入口。模型调用、标注方法、Prompt 初始化方法、分析方法都可以注册。

落地结果：

项目设置了 `user_hooks/` 包，里面包含：

- `llm_chat.py`
- `annotation_methods.py`
- `prompt_init_methods.py`
- `analysis_methods.py`
- `prompt_utils.py`

这样后端主体比较稳定，业务逻辑可以单独改。

## 交互 8：标注方案设计

我的提问：

```text
一个标注方案里面应该有哪些字段？我希望用户能选择 Prompt、知识库、模型、并发数和后台方法。
```

AI 帮助：

AI 建议标注方案包含方案名称、所属场景、Prompt 多选、知识库多选、错题集多选、调用模型、并发数量、后台方法名、Prompt 初始化类型。

落地结果：

项目实现了方案管理，并在 PRD 中定义了 `schemes` 和 `scheme_resources` 两类数据结构。

## 交互 9：批量标注任务

我的提问：

```text
我需要批量标注和全量标注。用户点开始后，后端应该排队执行，前端要看到进度。这个怎么做比较简单？
```

AI 帮助：

AI 建议使用 Python `threading + queue` 做任务队列，用 SSE 给前端推送任务状态。任务行可以有排队中、标注中、TP、FP、失败、取消等状态。

落地结果：

项目实现了 `annotation_tasks` 接口，支持创建任务、查看任务、停止未完成标注和 SSE 事件流。

## 交互 10：实时指标和 TP/FP 计算

我的提问：

```text
工作台上要显示准确率、F1、TP、FP 这些指标。第一版我想简单一点，用人工答案列和模型答案列比较。
```

AI 帮助：

AI 建议第一版先用“人工答案列 == 模型答案列”判断 TP，不相等判断 FP，TN/FN 先保留位置。准确率、精确率、召回率、F1 可以放到指标区。

落地结果：

标注工作台顶部加入了指标区，展示总数、未标注、已标注、排队中、标注中、TP、TN、FP、FN、准确率、召回率、F1 等指标。

## 交互 11：大表格性能优化

我的提问：

```text
Excel 文件可能很大，前端一次性加载会卡。怎么优化？我现在是新手，希望方案别太复杂。
```

AI 帮助：

AI 建议后端负责分页、搜索和筛选，前端只展示当前页。表格使用 Tabulator，长字段只显示预览，完整内容放到详情弹窗或抽屉里。

落地结果：

项目采用了远程分页和大字段截断预览。README 中也写了前端性能策略：搜索和分页由后端处理，工作台使用 Tabulator。

## 交互 12：行详情和分析功能

我的提问：

```text
点击某一行后，我想看完整原始数据、模型返回结果、Prompt 和分析结果。这个适合做弹窗还是侧边栏？
```

AI 帮助：

AI 建议使用右侧详情抽屉，适合数据工作台，不会打断用户浏览表格。详情里可以分为查看、标注结果、编辑、分析几个 Tab。

落地结果：

标注工作台实现了行详情抽屉，支持查看原始数据、标注结果、编辑 JSON、分析结果等内容。

## 交互 15：模型返回结果写回表格

我的提问：

```text
模型返回的是 dict，里面可能有 GPT4_标注、原因、置信度等字段。前端表格怎么展示这些新字段？
```

AI 帮助：

AI 建议把模型返回的完整 dict 保存下来，同时把 dict 的 key/value 投影到当前行最新结果里。新 key 可以自动追加成表格列。

落地结果：

项目设计了最新标注投影字段，标注方法返回的 dict 会更新当前行展示，同时历史任务行会保留记录。

## 交互 16：停止未完成标注

我的提问：

```text
用户点了全量标注后，可能发现选错方案了。停止按钮怎么处理比较合理？
```

AI 帮助：

AI 建议停止排队中的任务，正在运行的模型调用自然完成。这样实现简单，也能避免强行终止线程带来的异常。

落地结果：

项目实现了“停止未完成标注”，对应接口是 `/{task_id}/stop-unfinished`。

## 总结

这个项目中，AI 主要帮我完成了四类事情：

- 拆需求：把模糊想法拆成页面、接口、数据表和任务流程。
- 写代码：辅助实现 FastAPI 接口、SQLite 存储、前端页面和任务队列。
- 改体验：优化工作台布局、大字段展示、状态 badge、详情抽屉。
- 补文档：整理 README、PRD、设计规范和比赛展示材料。

我自己的主要工作是确定业务场景、选择功能优先级、检查页面效果、根据运行结果继续修改。AI 提供了实现方向和代码草稿，最终功能以本项目代码为准。
