# 本地数据标注与 Prompt 评测系统 — 业务 PRD

> 配套文档:[`nexus-design-system.md`](nexus-design-system.md)(视觉风格)、[`nexus-companies.html`](nexus-companies.html)(原型参考)。
> 本 PRD 描述**功能、流程、菜单、交互**。技术实现细节暂不展开。

---

## 1. 产品定位

面向大模型开发者的**本地化标注与 Prompt 评测工具**,核心解决:

1. 异构 Excel 数据的导入与合并
2. 动态 Prompt 模板的可视化构建
3. 多角色(多 Prompt)大模型并发评测
4. 在统一列表中比对人工答案 / 模型预测,做**指标统计、筛选分析、人工干预**
5. 通过**反复标注 + 调整 Prompt** 提升模型准确率

**典型场景**:SPN、IPRAN、泰国、印尼……(以电信项目编号或国家维度划分的标注集)。

**目标价值**:开发者能在同一个工作台里完成"上传数据 → 写 Prompt → 跑模型 → 看指标 → 改 Prompt → 再跑"的完整闭环,把模型准确率持续推高。

## 2. 术语表

| 术语 | 含义 |
| --- | --- |
| **场景 (Scene)** | 最外层业务隔离单元,如 SPN、IPRAN、泰国、印尼。仅需命名 |
| **数据集 (Dataset)** | 场景下的 Excel 文件;可单独标注,也可与同场景其他数据集合并为一份 |
| **字段映射** | 场景级配置,4 项规则,向下应用到该场景所有数据集 |
| **方案 (Scheme)** | 场景级,一组 Prompt 的组合 + 并发参数。单 Prompt 或多 Prompt 自由组合 |
| **Prompt** | 方案内的最小执行单元:角色名 + 模板文本 + 绑定模型 + 输出建议 |
| **角色名** | Prompt 的唯一标识(同方案内唯一),参与落盘列名拼接,如"初审"、"质检" |
| **任务 (Task)** | 一次"对若干行执行某方案"的运行实例。每次跑批 = 一个独立任务 |
| **指标条** | 标注工作台顶部一行直观统计:总数、已标、准确率、TP/TN/FP/FN 等 |
| **占位符** | Prompt 模板中的 `{{字段名}}`,运行时被替换为该行数据 |
| **GT (Ground Truth)** | 字段映射的"答案列",即人工标注的标准答案 |
| **Prediction** | 字段映射的"预测列",即模型最终落盘列 |
| **知识库片段** | 场景内手工维护的文本片段,Prompt 中以 `{{知识库.<名称>}}` 引用 |
| **错题 (ErrorEntry)** | 从标注结果反哺的一条错例记录,由用户在标注工作台选行 + 选列后系统自动生成 JSON 保存 |
| **错题集 (ErrorSet)** | N 条错题合并成的命名集合;Prompt 中以 `{{错题集.<名称>}}` 引用 |
| **错题本** | 场景内的"错题与错题集"管理页(Tab),非数据实体 |
| **UserHooks** | 仅由使用者编写的代码区,目前 4 个方法:标注、翻译、分析、Prompt 自定义组装 |

## 3. 菜单与功能总览(产品地图)

### 3.1 左侧边栏(6 项,扁平结构)

从上往下:

| 顺序 | 菜单 | 功能简述 |
| --- | --- | --- |
| 1 | **数据集管理** | 上传、管理 Excel 数据集;每个数据集独立配置字段映射;可勾选多份合并为新数据集 |
| 2 | **标注工作台 ⭐** | 核心战场。内含 3 个 Tab:**标注 / 方案管理 / 任务面板**(见 3.2) |
| 3 | **Prompt 管理** | 全局 Prompt 库,Prompt 卡片可被多个方案复用;支持按场景标签过滤 |
| 4 | **知识管理** | 全局知识片段库,Prompt 中以 `{{知识库.<名称>}}` 引用 |
| 5 | **错题集管理** | 标注反哺的错题与错题集库,Prompt 中以 `{{错题集.<名称>}}` 引用 |
| 6 | **模型管理** | 只读列表,展示 UserHooks 里 `models` 字典暴露的所有模型(见 §8.1) |

> **场景 (Scene) 的定位变化**:从原"最外层导航维度"降级为各资源(数据集 / Prompt / 知识 / 错题集)的**标签字段**。每个管理页顶部有一个「场景 ▾」筛选器,选定后只显示该场景下的资源。场景命名仍延续 SPN、IPRAN、泰国、印尼等业务标识。

### 3.2 标注工作台内的 Tab(3 个)

| Tab | 功能简述 |
| --- | --- |
| **标注 ⭐** | 选择数据集 + 方案 → 顶部指标条(14 项)→ 工具栏 + 数据表 → 单/多/全选跑批 → 详情抽屉 → 批量加入错题本 |
| **方案管理** | 创建 / 编辑方案。方案 = 选择已有的 Prompt 卡片组合 + 并发参数;一个 Prompt 可被多个方案引用 |
| **任务面板** | 该工作区所有跑批任务的进度、历史、失败重跑、取消 |

### 3.3 行级操作(标注工作台 → 每行右侧 ⋯ 菜单)

| 操作 | 行为简述 |
| --- | --- |
| **标注** | 立即用当前方案对该行触发一次新任务 |
| **详情** | 打开侧边抽屉,查看该行原始字段 + 渲染后 Prompt + 各角色返回 |
| **标注历史** | 同侧边抽屉,默认聚焦"历史"Tab,看该行历次任务的差异 |
| **编辑** | 编辑该行的原始字段值(改完后下次重跑会用新值) |
| **分析** | 调用 UserHooks.analyze 对该行数据做后处理(用户自定义实现,默认 mock) |
| **导出** | 导出该行的所有字段(原始 + 渲染 Prompt + 各角色结果)为 JSON |
| **删除** | 删除该行数据(红色,带二次确认) |

## 4. 业务对象模型(逻辑视角)

```
全局扁平资源(每个资源带可选的 scene 标签字段)
├── 数据集 (Dataset)            每个数据集独立配置字段映射(4 项规则)
│       └── 行 (DataRow)
├── Prompt 卡片                 全局复用,含 角色名 / 模板 / 处理模式 / 默认模型
├── 方案 (Scheme)               引用一组 Prompt 卡片 + 并发参数
├── 知识库片段                  被 {{知识库.<名称>}} 引用
├── 错题 (ErrorEntry)            由标注工作台保存而来(行 × 选定列 → JSON)
├── 错题集 (ErrorSet)            N 条错题合并并命名,被 {{错题集.<名称>}} 引用
└── 任务 (Task)                 N 个,带历史结果(PromptResult)

模型 (Model)                    来自 UserHooks 内的 models 字典,只读
场景 (Scene)                    标签维度,出现在以上各资源的 scene 字段里
```

### 关键约束

- Prompt 输出**不强制骨架**,但编辑器输入框底部自动给出"推荐输出 JSON 格式"建议
- 建议中**必须包含场景的答案列字段名**(由字段映射决定),作为模型输出 JSON 的一个 key
- 落盘时,模型返回 JSON 的所有 key 被展平为列:`[角色名]_<key>`(例如 `[初审]_thinking`、`[初审]_情感分类`)
- 一行只有当方案内**所有 Prompt 都成功落盘**才算 🟢 已完成
- 同一行同一方案可多次执行(每次 = 一个 Task),**表格默认显示最新**,抽屉可回溯历史

## 5. 核心业务流程

```
① 「数据集管理」上传 Excel → 选定场景标签 → 配置该数据集的字段映射
       │ (字段:显示列、参考列、答案列、预测列)
       │
② 「Prompt 管理」创建 Prompt 卡片
   - 写模板,使用 {{字段名}}、{{知识库.xxx}}、{{错题集.xxx}} 占位符
   - 选处理模式(自动 / 自定义)
   - 选默认模型(来自 UserHooks.models)
   - 底部输出建议自动带出"答案列"作为 JSON key
       │
③ 进入「标注工作台」→「方案管理」Tab
   - 新建方案,从全局 Prompt 库挑选 N 张卡片组合
   - 配置并发量
       │
④ 「标注工作台」→「标注」Tab
   - 选数据集 + 选方案
   - 顶部指标条实时反映当前数据状态
   - 全选 / 多选 / 筛选目标行 → 开始标注
   - 长任务期间,行级 / 顶部均有加载动效
       │
⑤ 看指标 + 看结果
   - 指标条点击 FP/FN/失败 → 联动筛选,定位有问题的行
   - 双击行 / 操作列详情 → 侧边抽屉看具体 JSON 与渲染 Prompt
       │
⑥ 反哺 Prompt(关键迭代闭环)
   - 选中错例行 → 批量操作栏「加入错题本」→ 弹窗选列保存为错题
   - 「错题集管理」整理散错题,合并为命名错题集
   - 回到 Prompt 编辑器,插入 {{错题集.<名称>}}
       │
⑦ 调整 → 再跑
   - 改 Prompt 文本 → 用同一方案对失败/错分子集重新标注
   - 反复迭代,直至准确率达标
       │
⑧ 导出
   - 顶部:整个数据集导出 Excel
   - 行级:单行导出 JSON
   - 抽屉:选中单元格内容导出
```

## 6. 详细功能规格

### 6.1 场景标签(横切维度)

- 场景作为一个**字符串标签字段**附在每个数据集 / Prompt / 知识 / 错题集上(创建资源时下拉选或输入新场景名)
- 各管理页顶部统一有「场景 ▾」筛选器,选定后只显示该场景下的资源;选「全部」时显示所有
- 没有"新建场景"独立入口;首次在某资源里输入新的场景名时自动登记到全局场景列表
- 设置页可重命名 / 合并 / 删除场景标签(批量影响所有引用该标签的资源)

### 6.2 数据集管理(侧边栏 1)

#### 列表页

| 字段 | 说明 |
| --- | --- |
| 名称 | 上传时的文件名,可重命名 |
| 场景 | 标签徽章(可点击筛选) |
| 行数 / 列数 | 解析后的统计 |
| 字段映射状态 | ✅ 已配置 / ⚠ 未配置 |
| 创建时间 | 相对时间 |
| 操作 | 预览 / 编辑映射 / 重命名 / 删除 / 进入标注工作台 |

#### 操作

| 功能 | 说明 |
| --- | --- |
| 上传 | 拖拽 / 点选上传 `.xlsx`,选场景标签后入库;失败要明确报错 |
| 合并 | 勾选多个数据集 → 「合并为新数据集」,生成新虚拟数据集(列取并集,缺失补 null);合并结果继承字段映射或重新配置 |
| 导出 | 当前数据集导出 Excel(原始列 + 所有展平的角色结果列) |
| 删除 | 二次确认,会级联清空该数据集对应的标注历史 |

> 数据不做自动去重(用户语料的多样性本身有价值)。

#### 字段映射(每个数据集独立配置)

进入数据集详情或点击「编辑映射」打开一屏配置:

1. **默认显示列**(多选):决定「标注工作台」列表默认显示哪些原始列
2. **标注参考列**(多选):决定 Prompt 中 `{{字段名}}` 可用的变量
3. **答案列**(单选,**关键**):Ground Truth;此列字段名会自动出现在 Prompt 编辑器的"输出建议"中,作为模型 JSON 输出的 key
4. **预测列**(单选):评测指标的对照列;可与某个 Prompt 角色的 `<答案列>` 自动联动

**异构 Excel 处理**:不同 Excel 列名集合不一致时,UI 显示并集供勾选;某些列可能仅存在于部分文件,运行时缺值即视为 null。

**合并数据集的映射继承**:从来源数据集中选一份作为模板,或重新配置。

### 6.3 Prompt 管理(侧边栏 3)

全局 Prompt 库,每个 Prompt 卡片是一个可复用资源,可被多个方案引用。

#### 列表页

| 字段 | 说明 |
| --- | --- |
| 角色名 | 用于在方案里识别,如"初审"、"质检" |
| 场景 | 标签徽章 |
| 处理模式 | 自动 / 自定义 |
| 默认模型 | 来自模型管理 |
| 引用方案数 | 该 Prompt 被多少个方案使用(只读) |
| 最近编辑时间 | 相对时间 |
| 操作 | 编辑 / 复制 / 删除 |

> 删除 Prompt 前若有方案在引用,会弹窗提示并列出受影响方案,确认后才执行。

#### Prompt 编辑器

```
┌────────────────────────────────────────────────────────┐
│ 角色名(必填,同方案内唯一,如"初审"、"质检")             │
│ 绑定模型(下拉,从模型管理读取)                          │
│ 处理模式(单选,关键)                                   │
│   ○ 自动占位符替换  (默认,推荐)                       │
│   ○ 自定义处理       (调用 UserHooks.init_prompt)     │
│ Prompt 模板(代码编辑器风格,大文本框,支持等宽字体)       │
│   - 自动模式:支持 {{字段名}}、{{知识库.<名称>}}、       │
│              {{错题集.<名称>}} 三类占位符               │
│     右侧侧栏列出"可用变量",点击光标处插入                │
│   - 自定义模式:模板原样传给 init_prompt,占位符语法可由│
│     使用者自行约定;侧栏隐藏                              │
│                                                        │
│ ┌── 输入框底部:输出格式建议(可折叠/一键插入)─────────┐│
│ │ 💡 建议在 Prompt 末尾加入以下输出要求:                ││
│ │                                                      ││
│ │ 请严格按以下 JSON 格式返回:                            ││
│ │ {                                                    ││
│ │   "thinking": "你的推理过程",                         ││
│ │   "<场景答案列名>": "你的判定结果"                    ││
│ │ }                                                    ││
│ │                                                      ││
│ │ [📋 复制]  [➕ 一键插入到 Prompt 末尾]                ││
│ └──────────────────────────────────────────────────────┘│
└────────────────────────────────────────────────────────┘
```

#### 处理模式说明(关键)

每个 Prompt 卡片独立选择处理模式:

**模式 A:自动占位符替换(默认)**

- 系统在每次调用模型前,对模板做正则替换:
  - `{{字段名}}` → 当前行该字段的值
  - `{{知识库.<名称>}}` → 匹配的知识库片段内容
  - `{{错题集.<名称>}}` → 匹配的错题集(展开为该集合下所有错题的 JSON 数组)
- 编辑器右侧侧栏列出所有可用变量,点击即插入(变量按类型分组:行字段 / 知识库 / 错题集)
- 适合大多数场景,零代码改动

**模式 B:自定义处理**

- 系统不替换占位符;把**原始模板 + 行数据 + 知识库 + 错题集**整体交给 `UserHooks.init_prompt`,由使用者自行拼装最终 Prompt
- 使用者可以:用 Jinja2、自定义 DSL、动态 few-shot、按字段值条件拼接……
- 系统拿到 `init_prompt` 的返回字符串直接喂给模型
- 适合复杂的提示词组装逻辑

> 同一个方案内,不同 Prompt 卡片可以分别选不同模式(例如"初审"用自动、"质检"用自定义)。

#### 输出建议规则

- **非强制**:用户可改写、删除或完全不使用 JSON 包裹
- 建议中的 `<场景答案列名>` 来自字段映射的「答案列」(例如答案列叫 `情感分类`,建议自动显示 `"情感分类": "你的判定结果"`)
- 答案列变更后,**建议文本自动联动**,但用户已写好的 Prompt 文本不会被改动(只更新提示卡)
- 系统不强制校验,但若模型返回 JSON 不含该 key,该行的"对照列"将取不到值,在抽屉中以红色提示

### 6.4 模型管理(侧边栏 6)— 只读

模型的真正注册由使用者在代码中维护(见 §8.1),前端只做**只读展示**与连通性测试。

| 字段 | 说明 |
| --- | --- |
| 模型 Key | 来自 `UserHooks.models` 字典的 key |
| 状态 | 🟢 可调用 / 🔴 抛异常 / ⚪ 未测试 |
| 最近调用时间 | 来自调用日志 |
| 最近调用耗时 | 平均 / P95 |
| 操作 | 🔧 测试连通性(发一个最简 prompt) |

**新增模型**:无 UI 入口。使用者在 `UserHooks` 类的 `models` 字典里加一项 `key: 调用方法`,**重启应用**后前端自动读取并出现。

**Prompt 卡片绑定模型时**:下拉框列出的就是 `models` 的所有 key,单选。

### 6.5 标注工作台 ⭐(侧边栏 2,沿用 Companies 表视觉)

#### 顶部指标条(关键,14 项)

分三组展示,实时计算:

**① 数据统计组(5 项,可点击筛选)**

| 指标 | 含义 | 颜色 |
| --- | --- | --- |
| 数据总数 | 当前数据集 / 视图下的总行数 | slate |
| 已标注 | 状态 = 🟢 已完成 的行数 | emerald |
| 未标注 | 状态 = ⚪ 未标注 的行数 | slate-light |
| 进行中 | 状态 = ⏳ 进行中 的行数 | amber |
| 失败 | 状态 = 🔴 失败 的行数 | rose |

**② 混淆矩阵组(4 项,可点击筛选)** — 基于「对照列 vs GT」

| 指标 | 全称 | 含义 |
| --- | --- | --- |
| TP | True Positive | 预测为正、实际为正 ✅ |
| TN | True Negative | 预测为负、实际为负 ✅ |
| FP | False Positive | 预测为正、实际为负 ❌ 误报 |
| FN | False Negative | 预测为负、实际为正 ❌ 漏报 |

**③ 评测指标组(5 项,只读百分比)** — 由 TP/TN/FP/FN 计算

| 指标 | 公式 | 业务含义 |
| --- | --- | --- |
| 准确率 Accuracy | `(TP + TN) / 总数` | 整体判对的比例 |
| 精确率 Precision | `TP / (TP + FP)` | 预测正里有多少是真的正(查准) |
| 召回率 Recall | `TP / (TP + FN)` | 实际正里被找到了多少(查全) |
| F1 分数 | `2·P·R / (P + R)` | 精确率与召回率的调和平均 |
| 特异度 Specificity | `TN / (TN + FP)` | 实际负里被正确识别的比例 |

**交互**

- 数据统计组与混淆矩阵组的指标块**点击即筛选**(例如点 FP 12 → 表格只剩 FP 行;再点一次取消)
- 评测指标组只读,鼠标 hover 显示完整公式 + 分子分母数值的 tooltip
- 指标随表格筛选 / 方案切换 / 对照角色切换**实时刷新**,数字滚动过渡不闪屏
- **多分类自动切换**:答案列出现 >2 种取值时,混淆矩阵组隐藏,评测指标组改为"各类别准确率"列表,按类别行数从大到小排列

#### 指标计算口径(基线规则)

1. **对照列的选取**:多 Prompt 方案下,在「场景设置」中指定某一个 Prompt 角色作为"主对照",其 `[角色]_<答案列>` 即评测对照列;若未指定,默认取方案中第一个 Prompt 的角色
   - 指标条右上提供"对照角色 ▾"下拉,可临时切换
2. **正类约定**:二分类时需指定正类。默认取答案列中**首次出现的值**作为正类;可在场景设置中显式声明
3. **多分类支持**:若答案列出现 >2 种取值,自动切换为"各类别准确率"展示(替代 TP/TN/FP/FN),按类别行数从大到小排列

#### 工具栏与视图 Tabs

- 预设 Tab:全部 / 未标注 / 已完成 / 失败 / 与 GT 不一致
- 自定义视图:Notion/Airtable 风格筛选器(字段 + 操作符 + 值,AND/OR 组合),可命名保存
- 筛选器支持的字段:任意基础列、`[角色]_<key>`、状态、GT、TP/TN/FP/FN 分类

#### 表格列规则

## 7. 长任务体验与加载动效

> **背景**:大模型单次调用从几十秒到几分钟都属正常。前端必须对"等待"做充分的视觉表达,避免用户以为卡死。

### 7.1 全局原则

1. **永不假死**:任何异步操作都要有 loading 状态(skeleton / spinner / progress bar)
2. **细粒度反馈**:能精确到行/单元格的 loading,就不要只在顶部转圈
3. **可继续操作**:任务运行期间,其他行可继续点击查看、编辑、再发起新任务
4. **失败要可见**:超时、网络错、解析失败均要明确提示,带"重试"按钮
5. **进度可见**:批量任务一定要有 `8 / 12 (66%)` 的精确进度

### 7.2 各位置的加载表达

| 位置 | 状态 | 表达 |
| --- | --- | --- |
| **数据行(标注中)** | 该行某 Prompt 正在调用 | 单元格内显示旋转 `⟳` + "标注中…" 文字;行左侧出现淡绿色进度条 |
| **数据行(已完成单 Prompt,等其他角色)** | 多 Prompt 部分完成 | 状态徽章变 🟡;已完成的单元格正常显示,未完成的继续 spinner |
| **数据行(失败)** | 该行某 Prompt 失败 | 单元格显示 ⚠ + 错误简述;hover 显示完整错误;操作菜单出现"重试" |
| **顶部指标条** | 任务进行中 | 「进行中」指标显示数字,带轻微脉冲动画 |
| **顶部全局任务 banner** | 有任务运行 | 可关闭的条形横幅:`当前任务 #ab12 · 8 / 12 · 跑批中`,点击跳任务面板 |
| **任务面板** | 任务卡片 | 进度条 + 已完成行数 / 总行数 + 启动至今的时长 |
| **侧边抽屉打开** | 数据加载中 | 抽屉内用 skeleton(灰条占位)代替骨架,加载完淡入 |
| **指标条切换/方案切换** | 重新计算 | 数字渐变过渡(数字从旧值滚动到新值),不闪屏 |
| **按钮 async** | "开始标注" 点击后 | 按钮变 disabled + 内含 spinner,完成或失败后还原 |
| **分析 / 翻译调用中** | 弹窗或 toast 内 | 中央 spinner + "处理中…"文字 |

### 7.3 视觉规范(对齐设计系统)

- 加载色:统一用品牌绿 `#22C55E`(loading bar 与 spinner)
- 失败色:统一用 `rose-500`
- 警告色(部分完成 / 解析问题):`amber-500`
- 动画时长:fade-in 200ms;spinner 旋转 1s/圈;进度条平滑过渡 300ms
- skeleton 用 `bg-slate-200` + `animate-pulse`

### 7.4 任务取消的及时性

- 用户点击"取消任务" → 立即标记任务状态为「正在取消」
- 已发出的 HTTP 请求不可逆,等当前批返回后整体停止
- UI 在「正在取消」状态显示橙色 spinner + "正在取消,等待当前批返回"

```python
    def _call_llama(self, prompt: str, role: str) -> str:
        ...
```

**前端如何用**:

- 应用启动时,前端读取 `UserHooks().models.keys()`,把这些字符串作为 Prompt 卡片「绑定模型」下拉的可选项,以及模型管理页的只读列表
- Prompt 卡片绑定模型 = 选其中一个 key(单选)
- 跑批时,系统通过 key 在 dict 里查到对应方法,调用 `fn(prompt, role)` 得到模型原始返回
- 使用者新增模型 → 改代码 → **重启应用** → 前端下拉自动出现新选项

**Mock 默认实现**:三个示例方法都做以下事情 — 随机 sleep 5~30s 模拟真实耗时,然后返回合理样例 JSON(如 `{"thinking": "推理过程示例", "情感分类": "正面"}`),偶发返回非法 JSON 触发解析失败路径,以便 UI 联调长任务与失败状态。

#### 8.1.2 三个钩子方法

| 方法 | 入参 | 返回 | Mock 默认行为 |
| --- | --- | --- | --- |
| **translate** | `text, target_lang="zh"` | 译文字符串 | 立即返回 `[已翻译·zh] <原文>` |
| **analyze** | `row_data: dict` | 任意 dict | 立即返回示例分析结果,如 `{"summary": "样例分析摘要", "key_signals": [...], "score": 0.85}` |
| **init_prompt** ⭐ | `prompt_template, row_data, knowledge, error_sets` | 最终 Prompt 字符串 | 默认实现把 `row_data` 序列化成 JSON 拼接到模板末尾后返回,既保证可跑通,又留下明显的"待自定义"痕迹 |

**`init_prompt` 入参细节**

- `row_data: dict` — 当前行所有原始字段 + 已有的角色结果 `[角色]_<key>` 列(以便参考)
- `knowledge: list[dict]` — 当前场景的知识库片段列表(`[{name, content, tags}]`)
- `error_sets: dict[str, list[dict]]` — key 是错题集名称,value 是该集合的错题 JSON 列表;使用者可按名称按需选取(例如只用某个特定集合)

**`init_prompt` 调用时机**:仅当某个 Prompt 卡片的「处理模式」选择了「自定义处理」时,系统才调用;否则走系统自带的占位符替换。

#### 8.1.3 启动即可演示

所有 Mock 实现**开箱即用**:无需用户改任何配置,启动后即可上传 Excel、配 Prompt、跑批、看指标、加错题集,演示完整闭环。
使用者要接入真实能力时,把对应方法体替换即可。

### 8.2 工程对使用者的零侵入承诺

- 工程其他文件**不会写入** UserHooks 所在文件
- UserHooks 文件顶部用醒目注释框标记"用户实现区"
- 应用启动时自动加载 UserHooks;**修改 `models` 字典或方法体后需重启应用生效**(避免运行中模型不一致)

### 8.3 代码结构规范(简要)

> 详细技术架构后续单独文档;这里只列原则。

- **分层清晰**:前端(展示 + 交互)/ 后端(业务逻辑 + 持久化)/ UserHooks(用户实现)三层职责单一,互不渗透
- **命名一致**:实体名英文(Scene、Dataset、Scheme、Prompt、Task、PromptResult),UI 文案中文
- **可维护**:
  - 每个业务模块独立目录(场景、数据集、方案、任务、指标)
  - 公共组件(Sidebar、按钮、徽章、抽屉、表格)抽离复用,与设计系统一致
  - 任务调度、占位符替换、JSON 解析等核心逻辑写单元测试
